# AI Content Studio — Domain Glossary

> Use these terms **exactly** in code, tests, commits, and conversation. If you encounter an undefined term during a session, add it here.

---

## Business Domain

| Term | Definition |
|------|-----------|
| **Business Profile** | The onboarding data a user provides: business type, products/services, target customers, brand identity. Stored in the `business_profiles` table. Drives ALL generation. |
| **Business Type** | Industry category: `restaurant`, `e-commerce`, `salon`, `gym`, `real_estate`, `fashion`, `freelancer`, `other`. Determines content templates and recommended posting schedules. |
| **Brand Kit** | The visual and verbal identity: colors (hex codes), tone (`professional` / `casual` / `playful` / `luxury`), logo URL, font preferences. Extracted from onboarding + logo upload. |
| **Content Calendar** | Auto-generated weekly/monthly schedule of planned content. Each slot specifies: date, time, platform, content type, and associated Asset Kit. |
| **Asset Kit** | A complete, ready-to-publish content unit: image or video + caption + hashtags + platform metadata. The atomic output of the generation pipeline. |
| **Content Slot** | A single position in the Content Calendar (e.g., "Monday 10am Instagram Post"). Gets filled with an Asset Kit. |
| **Content Type** | Category of content: `product_showcase`, `tip`, `behind_the_scenes`, `promo`, `testimonial`, `ugc_ad`, `seasonal`, `engagement`. |
| **Industry Template** | Pre-configured generation settings per Business Type. E.g., restaurant template emphasizes food photography + daily specials; gym template emphasizes motivational graphics + class promos. |

## AI / Generation Domain

| Term | Definition |
|------|-----------|
| **Agent Loop** | The core generation pipeline: **Plan → Retrieve → Route → Execute → Assemble → Publish**. Orchestrated by the Next.js backend. |
| **Plan Step** | AI analyzes the Business Profile + Content Slot → determines what content to create (content type, visual style, copy angle). |
| **Retrieve Step** | Queries the Brand Knowledge Base (pgvector) for relevant context chunks. Ensures brand-consistent output without dumping the entire profile into every prompt. |
| **Route Step** | Model Router selects the optimal Muapi model for each asset based on type, quality, cost, and speed. |
| **Execute Step** | Parallel API calls to Muapi (visual assets) and OpenAI (text assets). |
| **Assemble Step** | Combines generated media + text into a complete Asset Kit. Uploads media to R2, saves metadata to DB. |
| **Publish Step** | Optionally pushes the Asset Kit to YouTube / TikTok / Instagram via Muapi's social publishing API. |
| **Model Router** | Decision logic mapping asset types to AI models. E.g., product photo → `ai-product-photography`, social graphic → `flux-schnell`, UGC video → `creatify-lipsync`. |
| **Generation** | One invocation of an AI model producing one output (one image, one video clip, one text block). The atomic billing unit. |
| **Generation Credit** | Internal unit tracking AI usage per user per billing period. One generation = one credit. Different asset types may cost multiple credits. |
| **UGC Pipeline** | The 5-step User-Generated Content video pipeline: script → voiceover → lipsync → B-roll → assembly. Produces a talking-head style product review video at ~$0.91. |
| **B-Roll** | Secondary video footage (animated product photo) intercut with the main talking-head video. Adds visual variety to UGC ads. |
| **Brand Conditioning** | The practice of injecting **retrieved** Business Profile context (via RAG) into every AI prompt to ensure brand-consistent output. Uses the Brand Knowledge Base, not raw profile dumps. |
| **Asset** | A single generated file: one image, one video, or one audio clip. Multiple assets combine into an Asset Kit. |
| **Content Plan** | A proposed set of 5-7 content items for a given week, generated by the agent based on Business Profile + RAG context + industry best practices. User reviews, edits, and approves. Stored in `content_plans` table with status: draft → approved → generating → completed. |
| **Content Plan Item** | One entry in a Content Plan: `{ type, title, description, platform, scheduledDay, estimatedCredits }`. Becomes a calendar entry and Asset Kit after generation. |
| **Generation Queue** | The system that processes an approved Content Plan: iterates through items, calls the Agent Loop for each, tracks per-item status (pending → generating → completed → failed), enables parallel execution for images. |
| **Presenter** | A stock photo of a person used as the talking head in UGC videos. The agent pre-selects based on brand audience. User can change but doesn't have to. Replaces "avatar" to avoid confusion with user profile avatars. |
| **Prompt Template** | A parameterized prompt string with placeholders for Business Profile data. E.g., `"Create a {brandTone} Instagram post for a {businessType} featuring {productName}"`. |
| **Credits Display** | All credit usage shown to users as "X of Y left" (e.g., "15 of 50 remaining"). Never as dollar amounts, never as model names. Users see usage, not infrastructure. |
| **Industry Strategy** | A per-business-type content playbook loaded from `data/industry-strategies.json`. Defines which content types to create on which days (e.g., Restaurant: food photo Mon, BTS kitchen Wed, testimonial Fri). Falls back to a generic strategy for unlisted types. Fed into the ContentPlannerService. |
| **Seasonal Calendar** | A static JSON file (`data/holidays.json`) of major holidays by region (US, IN, UK, UAE, etc.) with name, date, and tags. Combined with user-defined `businessEvents` from the Business Profile. The ContentPlannerService checks for upcoming events within 14 days and injects them into the GPT prompt. GPT knows how to create themed content — we just tell it WHICH events are coming. |
| **Business Events** | User-defined milestones stored in `businessProfile.businessEvents` as `{ name, date, recurring }`. Examples: "Anniversary Sale, March 5, yearly" or "Summer Collection Launch, June 15, one-time". Captured during onboarding Step 6. |
| **Performance Feedback Loop** | The system where user feedback (thumbs up/down, regeneration rate, caption edit rate) is aggregated by `AgentEvalsService.getPerformanceInsights()` and fed back into `ContentPlannerService.proposePlan()` to bias future plans toward content types the user's audience prefers. |
| **Agent Eval** | Automated quality checks on the agent's output. Includes: `scorePlanQuality()` (0-100 score, auto-regenerate below 60), pipeline reliability metrics, and user feedback aggregation. Data stored in `generation_feedback` and `pipeline_logs` tables. |
| **Pipeline Log** | A row in the `pipeline_logs` table recording one step of a generation pipeline: step name, model used, duration_ms, success/failure, error message, cost. Used by `AgentEvalsService.getPipelineReliability()` and surfaced in the Admin panel (Phase 7). |
| **Plan Quality Score** | A 0-100 score computed by `scorePlanQuality()` before showing a proposed plan to the user. Checks: platform coverage, content type diversity, product rotation, seasonal relevance. Plans scoring below 60 are silently regenerated. |

## RAG / Knowledge Base Domain

| Term | Definition |
|------|-----------|
| **Brand Knowledge Base** | A per-user semantic vector store (pgvector on Neon) containing embedded representations of all brand data: products, brand guidelines, past generations, user edits, and industry knowledge. Queried during the Retrieve step of the Agent Loop. |
| **Embedding** | A 1536-dimensional numerical vector representation of a text chunk, generated by OpenAI `text-embedding-3-small`. Stored in the `brand_embeddings` table. Cost: ~$0.02 per 1M tokens (essentially free). |
| **Retrieve Step** | The second step in the Agent Loop (Plan → **Retrieve** → Route → Execute → Assemble → Publish). Queries pgvector with the generation request to find the most relevant context chunks for that specific generation. |
| **Cosine Similarity Search** | The vector search algorithm used to find embeddings most similar to the query embedding. pgvector provides this via the `<=>` operator. Returns top-k results ranked by relevance. |
| **Context Chunk** | A single piece of retrieved text from the Brand Knowledge Base (e.g., one product description, one brand guideline paragraph, one past caption). Multiple chunks are assembled into the prompt context. |
| **Auto-Embedding** | The automatic process of embedding new data into the Brand Knowledge Base. Triggered on: onboarding completion, product create/update, generation completion, user caption edits. The system learns from every interaction. |
| **pgvector** | PostgreSQL extension for vector similarity search. Enabled on Neon with `CREATE EXTENSION vector`. Stores embeddings as `vector(1536)` columns and supports cosine distance indexing. |
| **top-k** | The number of most-relevant context chunks retrieved per query. Default k=8. Higher k = more context but higher token cost and potential noise. |

## Social Publishing Domain

| Term | Definition |
|------|-----------|
| **Connected Account** | A user's linked social media account (YouTube channel, TikTok account, Instagram Business account) authorized via OAuth through Muapi. |
| **external_user_id** | The Clerk user ID passed to Muapi when connecting social accounts. Scopes OAuth tokens per end-user in the multi-tenant model. |
| **account_id** | Muapi's internal identifier for a connected social account. Integer, returned from the accounts API. Used in all publish calls. |
| **Connect URL** | A one-time OAuth URL generated by Muapi (`POST /api/v1/social/{platform}/connect-url`). Sent to the user's browser to authorize posting. |
| **Publish Job** | An async social publishing request. Returns a `request_id`. Status checked via polling (`GET /api/v1/predictions/{id}/result`) or webhook. |
| **Platform ID** | Muapi's numeric platform identifier: YouTube = `1`, TikTok = `2`, Instagram = `3`. |

## Billing Domain

| Term | Definition |
|------|-----------|
| **Plan Tier** | Subscription level. Tentative: Free, Starter ($19/mo), Pro ($49/mo), Business ($99/mo). Each tier has different generation limits and feature access. Final pricing TBD after computing server costs. |
| **Generation Limit** | Maximum generation credits per billing period per plan tier. |
| **Usage Meter** | Real-time counter of generation credits consumed in the current billing period. Displayed in the dashboard. |
| **Billing Period** | Monthly cycle starting from subscription creation date. Credits reset at period start. |
| **Overage** | When a user exceeds their generation limit. Behavior TBD: hard block vs. pay-per-extra. |
| **COGS** | Cost of Goods Sold — the Muapi + OpenAI API costs incurred to serve one user. Tracked via `X-MuAPI-Cost-USD` response headers. |

## Technical / Workflow Domain

| Term | Definition |
|------|-----------|
| **Slice** | An atomic unit of work. One Linear issue, one focused change, one checkpoint. Takes 1–4 hours. |
| **Phase** | A group of related slices delivering a major feature. One phase = one PR (or one Replit checkpoint series). |
| **Checkpoint** | A named save point in Replit (equivalent to a git commit). Created after each slice is complete. |
| **Session** | One continuous work period by an agent. Starts with reading protocol files. Ends with PROGRESS.md update + Linear sync. |
| **Session Log** | An entry in PROGRESS.md recording: date, slice worked on, files touched, decisions made, tech debt observed, blockers. Newest entry at top. |
| **Current Status** | The top section of PROGRESS.md. Overwritten each session. Contains: active phase, current slice, next action, blockers, files last touched. |
| **Acceptance Criteria** | GIVEN/WHEN/THEN statements in PLAN.md defining when a slice is "done". Uses RFC 2119 language (MUST, SHOULD, MAY). |
| **ADR** | Architecture Decision Record. Documents a significant technical decision, its context, alternatives considered, and consequences. Stored in `docs/adr/`. |
| **Agent Loop** | (see AI domain) Also used loosely to mean "the generation pipeline" in conversation. |
| **Feedback Loop** | The commands run after every change: `pnpm typecheck && pnpm test && pnpm lint`. |

## Admin Domain

| Term | Definition |
|------|-----------|
| **Admin Panel** | A separate `/admin` route group accessible only to users with `role = 'admin'`. Provides dashboards for users, revenue, costs, and account management. |
| **Admin Role** | A value in the `users.role` column. Values: `user` (default), `admin`. Checked by middleware on all `/admin/*` routes. First admin bootstrapped manually in DB. |
| **MRR** | Monthly Recurring Revenue. Sum of all active subscription prices. Primary revenue health metric. Sourced from Stripe, cross-referenced with `subscriptions` table. |
| **Churn Rate** | Percentage of paying subscribers who cancel per month. Calculated as: (cancellations this month / paying subscribers at start of month) × 100. |
| **Conversion Funnel** | The progression: Free signup → Onboarding complete → First generation → Paid subscription. Each stage tracked via PostHog events. |
| **LTV** | Lifetime Value. Average total revenue per user over their entire subscription lifetime. LTV = ARPU × (1 / churn rate). |
| **ARPU** | Average Revenue Per User per month. Total MRR / total paying subscribers. |
| **Cost Alert** | An admin dashboard flag raised when a user's COGS exceeds the revenue their plan generates. Indicates a money-losing customer. |

## Analytics Domain

| Term | Definition |
|------|-----------|
| **GA4** | Google Analytics 4. Used for landing page traffic analysis: page views, source attribution, bounce rate, CTA clicks. Script loaded only on public pages. |
| **PostHog** | Product analytics platform (Cloud free tier, 1M events/mo). Used inside the authenticated app for: feature usage, funnels, retention, session recordings. |
| **Custom Event** | A named analytics event tracked via `posthog.capture()`. Standard events: `generation_started`, `generation_completed`, `publish_completed`, `plan_upgraded`, `onboarding_completed`, `feature_used`. |
| **Session Recording** | PostHog's screen recording feature. Enabled for debugging user issues. Linked from admin user detail view. |
| **Sentry** | Error tracking and performance monitoring platform. Captures unhandled exceptions, API errors, and slow transactions in production. Alerts the team before users report issues. |
| **Error Boundary** | A React component that catches JavaScript errors in its child component tree, logs them to Sentry, and renders a fallback UI instead of crashing the page. Every page has one. |

## Email Domain

| Term | Definition |
|------|-----------|
| **Brevo** | Email service provider (formerly Sendinblue). Used for transactional emails, lifecycle drip campaigns, and newsletters. Single vendor for all email. |
| **Transactional Email** | Triggered by a specific user action: welcome email (signup), generation complete, credit warning, payment failed. Sent via Brevo API. |
| **Lifecycle Campaign** | Automated email sequences triggered by user behavior: onboarding nudge (24h after incomplete onboarding), reactivation (7 days inactive), churn prevention (before cancellation). Configured in Brevo workflow builder. |
| **Newsletter** | Product updates and feature announcements sent to all users. Managed via Brevo campaigns. Frequency: monthly or on major releases. |
| **Email Template** | A Brevo template with dynamic variables (user name, product name, etc.). Templates stored in Brevo dashboard, referenced by `templateId` in API calls. |

## Design Domain

| Term | Definition |
|------|-----------|
| **Design Read** | A one-line declaration of page kind + audience + vibe + aesthetic direction. Stated before any UI code is written. See `DESIGN.md` §0. |
| **Three Dials** | Three tunable values that control all design output: `DESIGN_VARIANCE` (layout experimentation), `MOTION_INTENSITY` (animation depth), `VISUAL_DENSITY` (information per viewport). See `DESIGN.md` §1. |
| **The Lila Rule** | Design directive: no AI-purple gradients, no random neon glows. Accent color is Emerald, base is Zinc. See `DESIGN.md` §3. |
| **Asset Kit Preview** | The right panel in the Generate page showing the generated image/video + caption + hashtags. The primary visual output of the app. |
| **Skeleton Loader** | A loading placeholder that matches the exact shape of the final content. Used instead of generic spinners. Required by `DESIGN.md` §7. |
| **Empty State** | A beautifully composed placeholder shown when a list or section has no data. Must include a CTA indicating how to populate. Required by `DESIGN.md` §7. |
